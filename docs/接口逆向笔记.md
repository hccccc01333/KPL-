## 一、核心接口

### 1.1 单场比赛数据接口

```
https://prod.comp.smoba.qq.com/leaguesite/battle/open?battle_id={battle_id}
```

**请求方法**：GET
**返回格式**：JSON（注意 Content-Type 是 `text/plain`，但内容是合法 JSON）

**必需 Headers**（缺一不可）：
```python
HEADERS = {
    "User-Agent": "Mozilla/5.0 (Linux; Android 6.0; ...) Chrome Mobile",
    "Referer": "https://pvp.qq.com/",
    "Origin": "https://pvp.qq.com"
}
```

**踩坑记录**：
- 不带 `Referer` / `Origin` 会返回 403（防盗链）
- User-Agent 用桌面端浏览器也能通过，但移动端的更稳

## 二、battle_id 编码规则（重大发现）

`battle_id` 不是随机字符串，是**有规律的复合 ID**：

```
battle_id = {战队A_id}_{局数_id}_{战队B_id}

例：1038107152_18_1742644777
    ↑          ↑   ↑
  战队A      第几局  战队B
```

### 推论

1. **不需要去赛程页面手工抠 URL**——只要知道两队 ID + 局数范围，就能生成所有 URL
2. **局数 id 的起始值因赛事不同**：
   - KPL 常规赛：起点未知（待验证）
   - **挑战者杯：从 10 开始**（用户经验）
   - 其他赛事：待验证
3. BO7 比赛局数 id 是连续的（10、11、12...16）

### 待验证的猜想

- ❓ 战队 ID 互换位置（`A_id_X_B_id` vs `B_id_X_A_id`）是否同一场？
- ❓ 同一赛事内不同 BO 系列的局数 id 是否全局递增？

> 这些可以在 `notebooks/00_接口探索.ipynb` 里做实验验证。

## 三、赛程接口（待逆向）

### 3.1 赛程页面 URL

```
https://pvp.qq.com/matchdata/schedule.html?league_id={league_id}
```

**`league_id` 编码规则（推测）**：
```
league_id = {年份}{赛事编号}

例：20260002
    ↑       ↑
  2026年   第 02 个赛事（疑为挑战者杯）
```

### 3.2 比赛详情页 URL

```
https://pvp.qq.com/matchdata/scheduleDetails.html?league_id={league_id}&match_id={match_id}
```

`match_id` 格式：`2026042501`（推测：年月日 + 当日序号）

## 四、JSON 字段结构

### 4.1 顶层结构

```json
{
  "message": "",
  "code": 200,
  "request_id": "...",
  "data": { ... }   // ⭐ 核心数据在这里
}
```

### 4.2 `data` 内部结构（已验证字段）

#### 比赛级
| 字段 | 类型 | 含义 |
|---|---|---|
| `battle_id` | string | 比赛唯一 ID |
| `status` | int | 比赛状态（待解码：1=未开始？2=已结束？） |
| `win_camp` | int | 哪方获胜（1=蓝方，2=红方） |
| `game_duration` | int | 比赛时长（**毫秒**） |
| `battle_seq` | int | BO 系列里的第几局 |

#### 战队级（`camp1` / `camp2` 各一份）
| 字段 | 类型 | 含义 |
|---|---|---|
| `team_id` | string | 战队 ID |
| `team_name` | string | 战队全名（如"成都AG超玩会"）|
| `team_abbreviation` | string | 缩写（如"AG"） |
| `team_icon` | string (URL) | 战队 Logo |
| `is_win` | bool | 是否获胜 |
| `kill_num` / `death_num` / `assist_num` | int | 团队 KDA |
| `kda` | float | 团队 KDA 数值 |
| `gold` | int | 团队总经济 |
| `push_tower_num` | int | 推塔数 |
| `kill_big_dragon_num` | int | 主宰击杀 |
| `kill_dark_tyrant_num` | int | 暗影主宰击杀 |

| `kill_shadow_dragon_num` | int | 影子主宰 |
| `kill_storm_dragon_king_num` | int | 风暴龙王 |

#### 选手级（`battle_player_list`，10 个 dict 的数组）

## 五、实时数据流（V2 阶段重点）

### 5.1 已发现现象

> **同一个 `battle_id`，在比赛进行中拉取，返回的字段值会变化。**

意味着腾讯接口提供的不只是赛后归档数据，还包括"当前累积值"快照。

### 5.2 待摸清的关键问题

1. **更新频率**：腾讯后台多久刷一次？前端拉取频率限制多少？
   - 假设方案：每 5 秒探测一次，对比相邻响应的 hash，统计真实变化间隔
2. **`status` 字段的含义**：
   - 比赛未开始时是什么值？
   - 进行中是什么值？
   - 已结束是什么值？
3. **进行中接口返回的字段是否完整**？
   - `battle_player_list` 在比赛中是否有数据？
   - `game_duration` 是不是当前已进行的时间？

### 5.3 V2 设计思路

```python
# 伪代码
while not match_ended:
    snapshot = fetch_battle(battle_id)
    save_snapshot(snapshot, timestamp=now)
    if snapshot['status'] == END_STATUS:
        break
    time.sleep(15)  # 每 15 秒一次（保守值）
```

## 六、反爬注意事项

1. **必须加随机延迟**：
   ```python
   time.sleep(random.uniform(1.5, 3.5))
   ```
2. **不要并发请求**：腾讯接口对单 IP 有阈值
3. **保存原始 JSON**：万一字段后续要扩展不用重爬
4. **遇到 403/429 立即暂停**：被识别要冷却 30 分钟以上
