## 步骤 6 · 特征工程（核心难点 ⭐）

### 思路

> **这是修复老项目最大 bug 的关键步骤**：老项目用比赛结束后才有的字段（经济、KDA）当特征预测胜负——这是数据泄漏。新项目要严格区分**赛前可知**和**赛后才知**。

### 赛前可知特征（合法）

| 特征类别 | 例子 |
|---|---|
| 历史交锋 | A vs B 历史胜率、近 5 场胜率 |
| 战队近况 | 各队近 10 场场均经济、场均 KDA、场均推塔 |
| 选手实力 | 各位置选手 KDA 加权均值 |
| 阵容相对强度 | 历史同英雄胜率（赛后才知 BP，但 BP 完成后即可用）|
| 元信息 | 是否主场、赛程密度（连续作战）、近期休息天数 |

### 赛后才知特征（**禁用**）

- 当场比赛的 KDA、经济、推塔数、伤害...
- 这些是"结果"不是"原因"

### 任务

新建 `notebooks/02_特征工程.ipynb`：

1. 构造"队伍历史窗口特征"：
   ```python
   # 对每场比赛，回溯该队伍此前 N 场的均值
   # 用 groupby + rolling
   ```
2. 构造"对位历史特征"：A vs B 历史交锋的胜率、平均经济差
3. 构造"选手特征"（用 players.csv 聚合）
4. **特征筛选**：相关性热力图、方差过滤
5. 保存特征矩阵 → `data/processed/features.csv`

### 产出

- ✅ 一张 `features.csv`，每场比赛一行，含 20-30 个赛前特征 + 1 个 label（哪方胜）
- ✅ 特征工程 notebook，包含每个特征的"为什么这么造"

### 踩坑提示

- **时序顺序很关键**：构造历史特征时一定要按 `match_date` 排序后再 rolling
- **避免 lookahead**：第 N 场的特征只能用前 N-1 场的数据
- **冷启动**：赛季前几场没有历史特征，直接舍弃或填中位数
