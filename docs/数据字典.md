# 数据字典

> 本文档记录 KPL 接口返回 JSON 的所有字段含义。
> **每发现新字段或语义变化，必须回来更新这里。**
>
> 数据来源：`https://prod.comp.smoba.qq.com/leaguesite/battle/open?battle_id=xxx`
>
> 示例样本：`data/raw/sample.json`（TTG vs KSG，2026 挑战者杯）

---

## 接口三层结构（重要）

```
GET /leaguesite/leagues/open                              ← 拿 league_id
GET /leaguesite/matches/open?league_id=xxx                ← 拿 match_id
GET /leaguesite/match/battles/open?match_id=xxx           ← 拿 battle_id
GET /leaguesite/battle/open?battle_id=xxx                 ← 拿单局完整数据 ← 本表对应这个
```

> ⚠️ `battle_id` 格式 `xxxA_局数_xxxB` 中 A/B 是战队 ID，但**第三段不是简单递增**——同一场 BO 系列里第二局的第三段会变。
> 结论：**不能批量构造 battle_id，必须从赛程接口拉**。

## 二、战队级（`camp1` / `camp2`）

> **共 17 字段**（截至 2026-05 探索）。

| 字段名 | 类型 | 含义 | 示例 | 赛前/赛后 | 备注 |
|---|---|---|---|---|---|
| `team_id` | str | 战队 ID | `10017` | 赛前 | 主键 |
| `team_name` | str | 战队全名 | `广州TTG` | 赛前 | |
| `team_abbreviation` | str | 战队缩写 | `TTG` | 赛前 | |
| `team_icon` | str | 战队 logo URL | http://... | 赛前 | 无分析价值 |
| `is_win` | bool | 这队是否赢 | `true` | **赛后** | |
| `kill_num` | int | 击杀数 | `8` | **赛后** | |
| `assist_num` | int | 助攻数 | `22` | **赛后** | |
| `death_num` | int | 死亡数 | `4` | **赛后** | |
| `gold` | int | 总经济 | `47042` | **赛后** | |
| `kda` | float | 战队整体 KDA | `7.5` | **赛后** | (k+a)/d |
| `push_tower_num` | int | 推塔数 | `5` | **赛后** | |
| `kill_tyrant_num` | int | 暴君（小龙）数 | `1` | **赛后** | |
| `kill_dark_tyrant_num` | int | 暗影暴君数 | `1` | **赛后** | |
| `kill_big_dragon_num` | int | "大龙"数（**字段语义存疑**） | `1` | **赛后** | ⚠️ 经回放对比，疑似为先知主宰+暗影主宰合计 |
| `kill_prophet_dragon_num` | int | 先知主宰数 | `1` | **赛后** | |
| `kill_shadow_dragon_num` | int | 暗影主宰数 | `1` | **赛后** | |
| `kill_storm_dragon_king_num` | int | 风暴龙王数 | `0` | **赛后** | |

### ⚠️ 字段语义异常（重要发现）

通过对比游戏回放发现：当**暗影主宰**被击杀时，`kill_big_dragon_num` 和 `kill_shadow_dragon_num` 会**同时 +1**。

推测：腾讯接口里 `kill_big_dragon_num` 是历史遗留字段，老版本只有先知主宰，新版本引入暗影主宰后没有重命名，而是把暗影主宰也算进去了。

**建模影响**：直接用 `kill_big_dragon_num` 会与 `kill_prophet_dragon_num + kill_shadow_dragon_num` 高度重复，做特征时应当只选其中一组。

## 三、选手级（`battle_player_list`，长度 10）

> **共 37 字段**（json_normalize 展平后）。
> 一行 = 一场比赛里某一位选手。

### 3.1 身份字段（赛前可知）

| 字段名 | 类型 | 含义 | 示例 |
|---|---|---|---|
| `team_id` / `team_name` | str | 所属战队 | `10017` / `广州TTG` |
| `camp` | int | 阵营 | `1`=蓝 `2`=红 |
| `position` / `position_desc` | int / str | 位置（数字编码 / 文字） | 待补 / `对抗路` |
| `actual_player_name` | str | 战队前缀+ID | `广州TTG.萝卜` |
| `player_name` | str | 选手 ID | `萝卜` |
| `player_icon` | str | 头像 | http://... |

> position_desc 取值：`对抗路` / `打野` / `中路` / `发育路` / `游走`

### 3.2 英雄字段（BP 阶段可知）

| 字段名 | 类型 | 含义 |
|---|---|---|
| `hero_id` | int | 英雄 ID |
| `hero_name` | str | 英雄中文名（无需查字典）|
| `hero_icon` | str | 英雄图标 |

### 3.3 战绩字段（**赛后才有**）

| 字段名 | 类型 | 含义 |
|---|---|---|
| `kill_num` / `assist_num` / `death_num` | int | KDA 三件套 |
| `kda` | float | 综合 KDA |
| `gold` | int | 个人经济 |
| `hurt_total` | int | 总输出 |
| `be_hurt_total` | int | 总承伤 |

| `be_hurt_by_hero_total` | int | 被英雄输出 |

| `hurt_to_hero_total_rate` / `be_hurt_by_hero_total_rate` | float | 对英雄输出/承伤占比 |
| `participation_rate` | int | 参团率（%）|

| 字段名 | 类型 | 含义 | 备注 |

## 四、BP 级（`bp_list`）

> 一场比赛 **20 条**记录（10 ban + 10 pick）。
> 每条 = 一次 ban 或一次 pick 操作。

| 字段名 | 类型 | 含义 | 示例 |
|---|---|---|---|
| `camp` | int | 操作阵营 | `1`=蓝 `2`=红 |
| `is_ban_or_pick` | int | **操作类型** | `0`=ban `1`=pick |
| `hero_id` | int | 英雄 ID | `503` |
| `hero_name` | str | 英雄中文名 | `狂铁` |
| `hero_icon` | str | 英雄图标 | http://... |
| `position` | int | 用 pick 后的位置 | `0`=对抗 `1`=打野 `2`=中路 `3`=发育 `4`=游走 |

### KPL BP 流程（按 list 顺序）

```
索引 0-3 ：第一阶段 ban  ← 蓝 ban / 红 ban / 蓝 ban / 红 ban
索引 4-9 ：第一阶段 pick ← 蓝 1 / 红 2 / 蓝 2 / 红 1
索引 10-13：第二阶段 ban  ← 红 ban / 蓝 ban / 红 ban / 蓝 ban
索引 14-19：第二阶段 pick ← 红 1 / 蓝 2 / 红 2 / 蓝 1（？）
```

> 顺序结论由 sample.json 一局比赛归纳得出，建议后面爬完更多数据再交叉验证。
