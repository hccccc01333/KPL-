## 二、模块职责

| 层 | 模块 | 职责 |
|---|---|---|
| 数据采集 | `src/crawler/` | 调用接口、保存原始 JSON、反爬控制、断点续爬 |
| 数据解析 | `src/parser/` | JSON → DataFrame，处理嵌套与缺失 |
| 数据探索 | `notebooks/01_EDA` | 摸数据，找洞察 |
| 特征工程 | `notebooks/02_特征工程` | 严格区分赛前/赛后特征 |
| 建模 | `notebooks/03_模型对比` + `src/modeling/` | 训练、评估、保存模型 |
| 模拟器 | `src/modeling/simulator.py` | BO7 蒙特卡洛模拟 |
| 演示 | `notebooks/04_预测演示` | 可读的业务故事 |
| 工具 | `src/utils/` | 日志、路径管理等 |

## 三、关键设计决策

### 决策 1：为什么不直接 SQL，而用 pandas？

**理由**：项目数据规模 100-200 场比赛，pandas 内存完全够。引入 SQLite 反而增加复杂度。
后续若需多表联查，可再引入 SQLite / 数仓层。

### 决策 2：为什么用 conda 而不是 venv？

**理由**：xgboost、pandas 等数据栈在 conda 上预编译好，pip 装可能要本地编译（慢且容易报错）。
特别是 Windows 系统，conda 的体验明显更好。

### 决策 3：为什么不用深度学习？

**理由**：
1. 数据规模有限（百局级），树模型更稳、可校准
2. 需要可解释因素（Dashboard「为什么领先」），树模型 + 特征解释更匹配
3. SHAP / 特征重要性便于业务沟通

### 决策 4：为什么严格区分赛前/赛后特征？

**理由**：避免用终局才知道的字段做「赛前预测」造成泄漏。
- ❌ 错误：用比赛结束后的经济、KDA 预测胜负
- ✅ 正确：只用历史与 BP 阶段可知信息，实时阶段只用当时快照

### 决策 5：BO7 模拟器为什么用蒙特卡洛？

**理由**：单场胜率之外，观众更关心系列赛比分分布。
通过蒙特卡洛模拟可输出比分概率，便于内容与复盘场景。

## 四、风险与对策

| 风险 | 对策 |
|---|---|
| 接口变更 | 保存原始 JSON，再次解析 / 重新逆向 |
| 反爬封禁 | 随机延迟 + 单 IP 节流 + 失败重试 |
| 数据规模太小 | 跨赛季合并（2024 + 2025 + 2026） |
| 模型过拟合 | 交叉验证 + 严格特征筛选 + clean holdout |
| 季后赛验证样本少 | 用历史决赛 / holdout 做回测 |
